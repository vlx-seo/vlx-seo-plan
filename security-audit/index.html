Executive Summary

This is a redacted public summary. Each finding lists the issue, severity, CWE, impact and recommendation. Specific code locations, exploit paths and proof-of-concept evidence are withheld and delivered separately to Visualogyx.

Visualogyx Inc engaged Software Craft CR to perform Application Security Testing of its new platform (a Next.js + Payload CMS rewrite replacing the legacy WordPress site). The assessment integrated two engagements: Part A — a white-box review of the application source code; and Part B — an authenticated assessment of the live development environment (gated behind AWS ALB + Cognito SSO).

The platform is, overall, well-engineered from a security standpoint: enforcing security headers, SSO-gated non-production environment, no secrets in source control, XSS-safe rich-text rendering, explicit access control on all content collections, no lead-PII persistence, and neutralized legacy WordPress paths. The assessment nonetheless identified 18 findings, headlined by one High-severity privilege-escalation issue.

Approach & Methodology

Tests were conducted against industry best practices (OWASP, SANS) and a 25-case Application Security Testing checklist, combining a white-box code review (Part A) with an authenticated grey-box assessment of the live site (Part B). Site testing was performed over the client's own authenticated session — the assessor did not handle or enter any credentials.

  1. Requirements Gathering · Threat Modelling · Manual Testing (input validation, authn/authz, business logic, session, error handling, data protection, security config).
  2. Automated Testing · API & Web Services Testing · Security Headers & SSL/TLS · Vulnerability Assessment.
  3. Reporting & Recommendations · Remediation Assistance · Reassessment.

Findings (Redacted)

Code locations, vulnerable parameters, exploit paths and proof-of-concept are omitted from this public summary.
O-01
Privilege escalation — an editor can self-promote to admin
HIGHOPENCodeCWE-915/269/285Ease: Medium
IssueThe CMS user collection permits a lower-privileged account to modify a privilege-controlling field on its own record, allowing self-escalation to administrator.
ImpactA low-privileged editor could obtain full administrative control (user management + write access to all content). Complete break of role separation.
RecommendationRestrict write access to the role/privilege field to administrators via field-level access control.
O-02
Content-Security-Policy in Report-Only mode
MEDIUMOPENCodeRuntimeCWE-693Ease: Hard
IssueThe Content-Security-Policy is delivered in report-only mode (not enforcing) and permits unsafe inline/eval script sources. Confirmed on the live environment.
ImpactCSP provides no active mitigation against script injection; the policy is effectively inert.
RecommendationMove to an enforcing CSP using nonces/hashes, add a violation-report endpoint, and remove unsafe-inline/unsafe-eval.
O-03
No rate limiting on public form endpoints
MEDIUMOPENCodeCWE-400Ease: Medium
IssueThe public contact/demo endpoints trigger outbound emails and notifications per request with no rate limiting (a limiter utility exists but is not applied).
ImpactPotential email flooding of arbitrary addresses and quota/notification abuse. Gated by SSO on dev; applies to the public production site.
RecommendationApply per-IP/per-email rate limiting (return HTTP 429) backed by a shared store.
O-04
Stored XSS via unescaped structured-data (JSON-LD)
MEDIUMOPENCodeCWE-79/116Ease: Medium
IssueCMS-controlled fields are embedded into inline structured-data script blocks without adequate output encoding, allowing script-context breakout.
ImpactA CMS editor could inject script that runs for all visitors of the affected pages (session/cookie theft). Author-trusted input → Medium.
RecommendationHTML-escape all values before embedding them in inline JSON-LD; centralize the encoding helper.
O-05
Weak CMS authentication configuration
MEDIUMOPENCodeCWE-287/521Ease: Medium
IssueThe CMS authentication uses defaults with no password-complexity policy, no email verification, no MFA and no explicit session/token expiration.
ImpactWeak/unverified credentials are permitted; widens the blast radius of O-01. Mitigated on dev by Cognito SSO; unhardened for any non-SSO deployment.
RecommendationEnforce password rules, email verification, login lockout and token expiration; restrict admin access to privileged roles.
O-06
Missing HSTS on load-balancer edge redirects
LOWOPENRuntimeCWE-319Ease: Hard
IssuePre-authentication redirect responses issued at the load-balancer edge do not carry a Strict-Transport-Security header, though the application does.
ImpactThe first hop is not HSTS-protected, leaving a narrow SSL-stripping window on initial navigation.
RecommendationSet HSTS at the load-balancer/edge so all responses (including redirects) carry it.
O-07
Draft-preview secret handling
LOWOPENCodeCWE-598/208/284Ease: Hard
IssueA static preview secret is transmitted in the URL and validated with a non-constant-time comparison; a collection identifier is not allow-listed.
ImpactSecret may leak via referrer/logs; endpoint can act as a slug existence oracle. Bounded — no document content is returned.
RecommendationMove the secret out of the URL, use constant-time comparison, allow-list the collection, and rate-limit.
O-08
CAPTCHA fails open in development mode
LOWOPENCodeCWE-693/1188Ease: Medium
IssueWhen the CAPTCHA secret is unset in development mode, verification passes automatically. It fails closed in production.
ImpactAny non-production deployment running in development mode silently disables CAPTCHA on all forms.
RecommendationGate the bypass behind an explicit opt-in flag; forward the client IP to the verification service.
O-09
Hard-coded third-party indexing key in source
LOWOPENCodeCWE-798Ease: Hard
IssueA third-party search-indexing key is hard-coded in source and committed to history.
ImpactSmall blast radius — the key is only honored for the owning domain; the endpoint is otherwise authenticated.
RecommendationMove the key to environment configuration, rotate it, and rename the public key file.
O-10
Verbose upstream error reflection + unhandled parse
LOWOPENCodeCWE-209Ease: Hard
IssueAn indexing endpoint reflects raw upstream error bodies to the caller and does not guard request-body parsing.
ImpactMinor third-party internals disclosure (authenticated endpoint) and an unhandled error path.
RecommendationLog upstream detail server-side, return a generic error, and guard parsing.
O-11
Insecure empty-string fallback for critical secrets
LOWOPENCodeCWE-1188Ease: Hard
IssueCritical secrets (signing secret, database URL, storage token) fall back to an empty string when unset instead of failing fast.
ImpactA missing environment variable could initialize the CMS with an empty signing secret, weakening token/cookie signing.
RecommendationFail fast (throw) at startup when a critical environment variable is missing.
O-12
Local test script logs a secret prefix
LOWOPENCodeCWE-532Ease: Hard
IssueA local diagnostic script prints a partial API-key prefix to standard output.
ImpactLocal script (not deployed); partial value only. Very low.
RecommendationPrint a boolean presence indicator instead of any key characters.
O-13
Two content collections are publicly readable
LOWOPENCodeCWE-284Ease: Easy
IssueTaxonomy and media collections are fully public via the API; media has no publish gating.
ImpactNon-sensitive marketing data; draft-only media could be enumerable. No PII. Blocked by SSO on dev.
RecommendationAcceptable to leave public; add read filtering if pre-launch assets must remain private.
O-14
Deprecated X-XSS-Protection value
LOWOPENCodeRuntimeCWE-693Ease: Hard
IssueThe legacy X-XSS-Protection header uses a value that is deprecated and ignored by modern browsers.
ImpactNegligible; legacy hardening nuance.
RecommendationDisable the header (value 0) once an enforcing CSP is in place.
O-15
Heading HTML assembled by string concatenation
LOWOPENCodeCWE-79 (potential)Ease: Hard
IssueSeveral UI components build heading HTML by string interpolation and inject it as raw HTML.
ImpactInputs are currently static/developer-controlled (not exploitable today); would become injectable if wired to dynamic data.
RecommendationRender dynamic emphasis as framework nodes rather than injecting raw HTML.
O-16
Outdated development-only dependency
LOWOPENCodeCWE-1035Ease: Hard
IssueA dev-only headless-browser dependency (used by scraping scripts) is several major versions behind.
ImpactLimited to the developer machine; not shipped in the production bundle.
RecommendationUpgrade the dependency and run a dependency audit.
O-17
Infrastructure disclosure via Server header
LOWOPENRuntimeCWE-200Ease: Easy
IssueEdge responses disclose the load-balancer technology via the Server header.
ImpactMinor fingerprinting aid.
RecommendationSuppress or normalize the Server header where possible.
O-18
Edge auth cookie without SameSite attribute
LOWOPENRuntimeCWE-1275Ease: Hard
IssueThe load-balancer authentication cookie is set Secure and HttpOnly but without an explicit SameSite attribute.
ImpactRelies on the browser default; minor CSRF-surface nuance on the auth flow.
RecommendationSet SameSite explicitly on the edge session cookie.

Positive Controls

  • ✅ Entire development environment gated behind managed SSO (AWS ALB + Cognito).
  • ✅ No secrets committed to source control; credentials sourced from environment configuration.
  • ✅ The API exposes no authentication secrets (no password hashes/salts).
  • ✅ No lead/PII persistence — contact/demo submissions are notification-only.
  • ✅ Explicit access control on all content collections; drafts hidden from anonymous readers.
  • ✅ CMS rich-text rendered safely (framework elements, not raw HTML).
  • ✅ Legacy CMS paths neutralized (return HTTP 410 Gone).
  • ✅ No SQL/command injection, open redirect or SSRF observed; CAPTCHA verified server-side.
  • ✅ No confidently-unpatched dependency CVE; framework version covers the notable 2025 advisories. Recommend a routine dependency audit.
7

Conclusion

The consolidated assessment found a total of 18 gaps: 0 Critical, 1 High, 4 Medium, 13 Low, all Open. The single High-severity issue (privilege escalation) should be remediated before launch, as it defeats CMS role separation. The Medium set (CSP enforcement, form rate limiting, structured-data XSS, authentication hardening) should follow.

The migration to the new platform behind managed SSO closed or neutralized most legacy findings and introduced strong baseline controls. Full technical detail, code references and proof-of-concept are provided in the private report delivered to Visualogyx.
