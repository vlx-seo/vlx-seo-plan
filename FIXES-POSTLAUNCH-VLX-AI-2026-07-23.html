Proposed task

In-site cache-management tool

Let editors purge/invalidate CloudFront (and inspect cache state) from the site's own admin, without the AWS CLI or personal AWS credentials.

🎯
Why it's worth building
Today, every deploy or content fix stays invisible until someone invalidates CloudFront by hand on the CLI. That friction is exactly why "already-deployed fixes" look unapplied — see Why caching bites below (it's the root cause of #1).
🧩
MVP scope
Auth-protected action in the Payload admin to trigger a CloudFront invalidation: full purge (/*) or by path (e.g. /blog/*). Ideally a "purge this page" button in each post/page editor. Show the last invalidation status and a URL's current cache headers.
🔐
Security
AWS SDK with a least-privilege IAM role (cloudfront:CreateInvalidation / GetInvalidation on the one distribution). Never expose credentials client-side. Log who triggers each purge.
💸
Cost / rate-limit
Invalidations bill after the free tier (1,000/month) — avoid looped full purges; prefer path-scoped invalidations.
↪️
Redirect policy (related)
permanent: true (301) is cached aggressively; use 302 for rules that may change, so browser caches aren't poisoned.

Priority: medium. Doesn't block launch, but removes recurring operational friction and the "fixed-but-not-visible" risk.

Root-cause analysis

Why caching is affecting the site

Two caching layers explain the symptoms observed (fixes not showing; "every blog page redirects home" on one machine).

🌐
1. CDN (CloudFront) — Cache-Control: s-maxage=31536000
Pages are cached at the CDN for up to 1 year. After a deploy, CloudFront keeps serving the old HTML until an explicit invalidation. Consequences: deployed fixes "don't show" until purged; and it's the cause of #1 — the code already returns 404 for out-of-range pages, but the edge serves a cached 200.
🧭
2. Browser — permanent: true redirects (HTTP 301)
Browsers cache 301s indefinitely. An earlier pagination redirect (documented at next.config.ts:905, a ?page= variant that broke pagination) got cached by any browser that hit it, so that browser keeps redirecting /blog/page/N/ → /blog/ locally even though the server is now correct. In a clean/incognito browser, pagination works (200, no redirect).
💡
Takeaway
Caching doesn't "break" the site — it desyncs what a user sees from the server's real state. A cached 301 can't be un-cached from the server side (the user must clear cache or wait for expiry). Hence the value of an accessible invalidation tool and a more careful redirect policy (302 for temporary rules).
After the PR merges

Deploy checklist

A · Ship

  • Merge the PR to master → auto-deploy to AWS ECS (prod).
  • Invalidate CloudFront (aws cloudfront create-invalidation --distribution-id <ID> --paths "/*") — without this the CDN keeps serving old HTML.

B · Verify

  • curl -I https://vlx.ai/digital-inspections-software/oil-gas/ → 301 to /oil-and-gas/.
  • Blog pagination pages return 200 and show distinct posts; out-of-range pages return 404.

C · SEO propagation

  • Resubmit sitemap.xml in GSC + IndexNow; request indexing of /oil-and-gas/.
  • Validate schemas (BlogPosting, SoftwareApplication, AggregateOffer, CollectionPage) in the Rich Results Test.
  • Refresh the OG image cache in the Facebook Sharing Debugger / LinkedIn Post Inspector.
  • Confirm in GA4 Realtime that SPA navigations now register as pageviews.
